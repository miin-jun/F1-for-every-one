# For everyOne — 백엔드 작업 명세서

> 화면 설계서(SCR-*) 기준으로 도출한 백엔드 작업 목록입니다.
> 이 문서는 본인의 작업 리스트이자, 팀과 분담을 논의할 때 쓰는 협의 자료로 활용하세요.

---

## 0. 전체 그림 한 장 요약

```
[프론트 화면]                [백엔드가 만들어야 할 것]
─────────────────────────────────────────────────────
시작/로그인/회원가입   ──▶  Auth API + 이메일 인증 시스템
비밀번호 찾기/변경     ──▶  Password API
채팅(텍스트/음성)      ──▶  Chat API + LLM 연동 + STT 연동
대화 히스토리(사이드바) ──▶  ChatSession/Message CRUD
내 정보/회원탈퇴       ──▶  User API
```

총 **약 18개의 API 엔드포인트**, **4개의 DB 테이블**, **2개의 외부 연동**(LLM, 이메일 발송)이 필요합니다.

## 1. DB 모델 설계

### 1-1. User (회원)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | PK | 자동 증가 또는 UUID |
| email | VARCHAR, UNIQUE | 로그인 ID |
| password_hash | VARCHAR | bcrypt 또는 argon2 해시 |
| created_at | DATETIME | 가입 일시 |
| updated_at | DATETIME | 정보 수정 일시 |
| is_active | BOOLEAN | 탈퇴 시 False (소프트 삭제용, 선택) |

### 1-2. EmailVerification (이메일 인증)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | PK | |
| email | VARCHAR | 인증 대상 이메일 |
| code | VARCHAR(7) | 7자리 인증코드 |
| purpose | ENUM | `signup` / `password_reset` 구분 |
| expires_at | DATETIME | 발송 시각 + 3분 |
| verified | BOOLEAN | 인증 완료 여부 |
| created_at | DATETIME | |

> 같은 이메일로 재요청 시 기존 코드는 무효화하거나 가장 최근 것만 유효 처리.

### 1-3. ChatSession (대화 세션)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | PK | |
| user_id | FK → User | |
| title | VARCHAR | 첫 질문에서 자동 생성 또는 첫 메시지 일부 |
| created_at | DATETIME | |
| updated_at | DATETIME | 마지막 메시지 시각 — 사이드바 정렬 기준 |

### 1-4. ChatMessage (대화 메시지)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | PK | |
| session_id | FK → ChatSession | CASCADE 삭제 |
| role | ENUM | `user` / `assistant` |
| content | TEXT | 메시지 본문 |
| created_at | DATETIME | 말풍선의 시각 표시에 사용 |

## 2. API 엔드포인트 목록

### 2-1. 인증 / 회원가입 (REQ-AUTH-001, REQ-USER-001)

| 메서드 | 경로 | 화면 | 요청 | 응답 |
|---|---|---|---|---|
| POST | `/api/auth/email/check-duplicate` | 회원가입(11/12) | `{email}` | `{available: bool}` |
| POST | `/api/auth/email/send-code` | 회원가입(14), 비밀번호찾기(2) | `{email, purpose}` | `{expires_at}` |
| POST | `/api/auth/email/verify-code` | 회원가입(15), 비밀번호찾기(3) | `{email, code, purpose}` | `{verified: bool}` |
| POST | `/api/auth/signup` | 회원가입(16) | `{email, password, terms_agreed, privacy_agreed}` | `{user_id}` |
| POST | `/api/auth/login` | 로그인(6) | `{email, password}` | `{access_token, refresh_token}` |
| POST | `/api/auth/logout` | 채팅 메인(4-1) | (토큰) | `{success}` |

### 2-2. 비밀번호 (REQ-USER-004, REQ-USER-005)

| 메서드 | 경로 | 화면 | 요청 | 응답 |
|---|---|---|---|---|
| POST | `/api/auth/password/reset` | 비밀번호 찾기(6) | `{email, code, new_password}` | `{success}` |
| POST | `/api/users/me/password/verify` | 비밀번호 변경(2) | `{current_password}` | `{matched: bool}` |
| PUT | `/api/users/me/password` | 비밀번호 변경(4) | `{current_password, new_password}` | `{success}` (성공 시 자동 로그아웃) |

### 2-3. 회원 정보 / 탈퇴 (REQ-USER-002, 003)

| 메서드 | 경로 | 화면 | 요청 | 응답 |
|---|---|---|---|---|
| GET | `/api/users/me` | 채팅 메인(4-1) | (토큰) | `{email, created_at}` |
| POST | `/api/users/me/withdraw/verify` | 회원탈퇴(2) | `{password}` | `{verified: bool}` |
| DELETE | `/api/users/me` | 회원탈퇴(2/네) | (토큰) | `{success}` (대화 내역 전체 삭제) |

### 2-4. 채팅 (REQ-CHAT-001)

| 메서드 | 경로 | 화면 | 요청 | 응답 |
|---|---|---|---|---|
| POST | `/api/chat/sessions` | 새 대화(1) | (토큰) | `{session_id}` |
| POST | `/api/chat/sessions/{id}/messages` | 메시지 전송(4) | `{content}` | `{user_message, assistant_message}` 또는 SSE 스트리밍 |
| POST | `/api/chat/transcribe` | 음성 입력(6/7) | multipart audio file | `{text}` (STT 결과) |
| GET | `/api/chat/suggestions` | 채팅 메인(3) | - | `[{question}, ...]` 랜덤 N개 |

### 2-5. 대화 히스토리 (REQ-HIST-001, 002)

| 메서드 | 경로 | 화면 | 요청 | 응답 |
|---|---|---|---|---|
| GET | `/api/chat/sessions` | 사이드바(2) | (토큰) | `[{id, title, updated_at}, ...]` updated_at 내림차순 |
| GET | `/api/chat/sessions/{id}/messages` | 사이드바 클릭(2) | (토큰) | `[{role, content, created_at}, ...]` |
| DELETE | `/api/chat/sessions/{id}` | 히스토리 삭제(2-1) | (토큰) | `{success}` |

## 3. 외부 연동 / 별도 검토 필요

### 3-1. 이메일 발송
- 로컬 개발 단계에선 콘솔 출력 또는 Mailtrap 같은 테스트 서비스로 시작.
- 실제 발송은 SMTP(Gmail SMTP, AWS SES, SendGrid 등) 중 팀에서 결정 필요.

### 3-2. LLM (FIA/F1 답변)
- OpenAI API, Claude API, 또는 자체 RAG 파이프라인 중 선택.
- FIA 규정 PDF/문서를 벡터 DB에 임베딩 → 검색 + 답변 생성(RAG) 구조가 도메인 정확성 면에서 유리.
- **이 부분이 프로젝트의 핵심 가치 영역** — 별도 시간 배분 필요.

### 3-3. 음성 인식 (STT)
- OpenAI Whisper API, Google Speech-to-Text, 또는 브라우저 Web Speech API.
- 브라우저 Web Speech API를 쓰면 백엔드 부담 없음(프론트에서 처리) → 팀과 결정 필요.

## 4. 권장 작업 순서 (1인 기준 약 2~3주)

### Week 1: 토대 + 인증
1. 프로젝트 초기 세팅 (FastAPI/Django 결정, DB 연결, 폴더 구조)
2. User 모델 + 회원가입 API + 로그인 API + JWT 인증 미들웨어
3. EmailVerification 모델 + 인증코드 발송/검증 API (먼저 콘솔 출력으로 시작)
4. 비밀번호 찾기 / 변경 / 탈퇴 API

### Week 2: 채팅 본체
5. ChatSession / ChatMessage 모델
6. 세션 생성 + 히스토리 조회/삭제 API
7. LLM 연동 (가장 단순하게 OpenAI API 호출부터)
8. 메시지 송수신 API

### Week 3: 마감
9. 음성 입력 처리 (선택에 따라 구현 또는 프론트에 위임)
10. 추천 질문 API
11. 프론트와 통합 테스트, 버그 수정
12. 배포 (Docker + 클라우드)

## 6. 팀 분담 시 제안 분할안

| 묶음 | 포함 작업 | 난이도 |
|---|---|---|
| **A. 인증 도메인** | 회원가입, 로그인, 이메일 인증, 비밀번호, 탈퇴 | 중 |
| **B. 챗 도메인** | 세션/메시지 CRUD, LLM 연동, 추천 질문 | 상 |
| **C. 인프라/공통** | 프로젝트 세팅, JWT 미들웨어, 이메일 발송 모듈, STT, 배포 | 중 |

A/B/C로 나누면 3인 분담, A+C / B 식으로 나누면 2인 분담이 자연스럽습니다.
